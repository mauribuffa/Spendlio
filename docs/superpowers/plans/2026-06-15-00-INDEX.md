## Pre-flight decisions for the fresh session (decide once, up front)

These came out of plan authoring; resolve them before executing so plans stay consistent.

### 1. Test runner for `apps/api` and `apps/web` — currently NONE

Only `packages/{contracts,core,ui}` have Vitest. The slice plans that touch the backend
hit this and each proposed something slightly different. **Pick one policy and apply it
everywhere:**

- **Recommended (surgical):** keep `apps/api` and `apps/web` test-runner-free.
  - **Pure logic is tested where the runner already exists:** account-balance + FX math →
    `packages/core` (Plan 01); settlement-netting behavior → `packages/core` (Plan 04);
    all new Zod schemas → `packages/contracts` (Plans 04, 06).
  - **API endpoints** are verified with the documented **`curl` smoke** in each plan
    (including a cross-user isolation check that `user_id` scoping holds).
  - **Web pages** are verified with `pnpm --filter web typecheck` + a **manual smoke**
    against the running stack (`pnpm dev` / mprocs).
  - Where a plan step says "add `vitest.config.ts` to apps/api|web", **skip it** under this
    policy and use the curl/typecheck fallback the plan also documents.
- **Alternative (more coverage):** add Vitest to `apps/api` **once** as a tiny foundational
  task, then Plans 04 and 06 get real controller unit tests (with a faked `db`), and add a
  web Vitest config for the small pure helpers in Plans 02/03. More setup; only do this if
  you want API/web tests as a standing capability.

Default to **Recommended** unless the lead says otherwise. Either way, the *pure domain
logic* tests in `core`/`contracts`/`ui` are non-negotiable and already runnable.

### 2. Account balance currency assumption (Plan 01)

Each account has one currency, but transactions also each carry a currency. Plan 01 sums
raw transaction amounts per `accountId` and treats the net as the account's currency (no
per-transaction re-conversion). Confirm this matches the seed/intended model before building.
If accounts can hold mixed-currency transactions, the math needs revisiting.

## Reconciled findings (spec said X; reality is Y)

- **Receipts → "create transaction from receipt" is NOT supported by the backend.** The
  receipts controller has no link/convert route and nothing sets `Receipt.transactionId`
  from the API. Plan 02 **omits** that action and documents it. (Spec Slice B mentioned it as
  a maybe — treat as future work, possibly a worker/endpoint addition.)
- **`CreateSettlementInput` already exists** in `packages/contracts/src/split.ts`. Plan 04
  only adds a confirming test, not a new schema.
- **Balances already net settlements** — `SplitsService.balances()` already subtracts
  settlements with `status='settled'`. No `core/balances.ts` change needed. **Load-bearing
  detail:** recorded payments must be inserted with `status:'settled'` + `settledAt`, or they
  won't move balances (the column defaults to `pending`). Plan 04 handles this.
- **`UpdateUserInput` already exists** but is a broad `CreateUserInput.partial()`. Plan 06
  **narrows** it to `pick({ name, defaultCurrency }).partial()` so Settings can't mutate
  email/locale/timezone. (This is the one design choice in Plan 06 worth a second look.)
- **Supported-currency list:** there is no currency enum; `CURRENCY_DECIMALS` in
  `packages/contracts/src/money.ts` lists the 17 supported currencies. Plans 01/06 reuse
  `Object.keys(CURRENCY_DECIMALS)` for the currency `Select` and treat conversion as
  approximate, returning `null` (UI shows "—") when no FX pair connects.
- **Account balance math is NOT in `core/balances.ts`** (that file is the who-owes-whom split
  graph). Plan 01 adds a separate `packages/core/src/accounts.ts`.
- **`SegmentedControl` is a client component** — pages using tabs (Accounts, Recap) split into
  a server page that does the `safe()` fetch + a `'use client'` child that owns tab state.
