# 13 · Internationalization (i18n)

> Designing for multiple languages, locales, and regions from the start. The expensive mistakes (hard-coded strings, locale ≠ currency confusion, ignoring plurals/RTL) are cheap to avoid early and painful to retrofit.

## The mental model: three separate axes
People conflate these; keep them distinct.

| Axis | Question | Driven by |
|---|---|---|
| **Language** | what words do I read? | the user's chosen **locale** (e.g. `es-AR`, `en-US`) |
| **Formatting** | how are numbers/dates shown? | the **locale** (1,000.50 vs 1.000,50; MM/DD vs DD/MM) |
| **Currency** | what money do I hold? | the **account/transaction currency** + the user's base currency |

**Key rule: locale ≠ currency.** A user can read Spanish (`es-AR`) but track money in USD, or read English while holding pesos. We store them **separately**: `user.locale` and `user.defaultCurrency`. Never infer one from the other. (This is why `formatMoney` takes both an amount-with-currency *and* a locale — see `12-currency-and-fx.md`.)

## What gets internationalized
1. **UI strings** — every visible label/sentence comes from a **message catalog**, never a hard-coded literal.
2. **Numbers, dates, currency** — formatted with the `Intl` APIs by locale (we already do for money).
3. **Plurals & gender** — "1 expense" vs "3 expenses"; handled by ICU MessageFormat, not string concatenation.
4. **Direction (RTL)** — Arabic/Hebrew flip the layout; the design must be direction-agnostic.
5. **User content** — transaction notes, people's names: user data, shown as-is (not translated), but must render any script.

## Where translations live (and where they don't)
- **In the apps, not in `contracts`/`core`.** `contracts` holds *keys/enums* (e.g. `category.key = 'groceries'`); the human label `"Groceries" / "Comestibles"` is a **translation in the app**, looked up by key. Domain logic stays language-free.
- **Default categories:** store the stable `key`; translate the label in the UI catalog (don't store English labels as the source of truth).
- **Catalog format:** ICU MessageFormat JSON per locale, e.g. `apps/web/messages/en.json`, `es.json`.

## Tooling (🟡 proposed)
- **Web (Next.js App Router):** **`next-intl`** — server-component friendly, ICU support, locale routing (`/en`, `/es`). Alternatives: `next-i18next`, Next's built-in i18n.
- **Mobile (later):** `i18next` / `react-i18next` or the platform's native i18n.
- **One shared catalog source** can be generated, but each app owns its rendering. Keep keys consistent across apps.

## ICU MessageFormat — the one pattern to learn
Don't build sentences by concatenation; it breaks in other languages (word order, plurals, gender).
```json
{
  "owedSummary": "You're owed {amount} from {count, plural, one {# person} other {# people}}",
  "addExpense": "Add expense"
}
```
```ts
t('owedSummary', { amount: formatMoney(money, locale), count }) // plural rules are per-locale
```
`Intl.PluralRules`, `Intl.NumberFormat`, `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat` do the locale-specific heavy lifting — use them, never hand-roll.

## Dates, times, timezones
- Store timestamps as UTC `timestamptz`; **format in the user's locale + timezone** at the edge with `Intl.DateTimeFormat`.
- Relative dates ("Today", "Hace 2 días") via `Intl.RelativeTimeFormat`.
- A user's timezone is its own setting (`user.timezone`), separate from locale.

## RTL (right-to-left)
- Use **logical CSS** (`margin-inline-start`, `padding-inline`, `inset-inline`) not `left/right`, and flexbox/grid which mirror automatically.
- Set `dir="rtl"` on `<html>` for RTL locales; icons that imply direction (chevrons, the split arrows) flip.
- The design system already leans on flex/grid + tokens, so it's mostly RTL-ready; audit any hard-coded `left/right`.

## Practical rules (house style)
- **No string literals in components** — every label is `t('key')`. A lint rule can enforce this.
- **Keys are namespaced** (`accounts.title`, `split.evenly`) and stable; English copy is just the `en` value, not the key.
- **Pseudo-localization** in dev (`[!!Àdd éxpénsé!!]`) surfaces hard-coded strings and truncation early.
- **Don't translate** brand name "Spendlio", currency codes, or user content.
- **Number/money input** must accept locale formats (comma vs dot decimal) — parse with care.

## Phasing
- **Now (foundation):** keep `user.locale` + `user.timezone` in the schema; store category **keys** not labels; format all money/dates via `Intl`; no hard-coded strings in new UI.
- **When we localize:** add `next-intl` + `en`/`es` catalogs; extract existing strings; add locale routing; RTL audit.
