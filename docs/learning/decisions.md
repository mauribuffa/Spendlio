# Decisions log (ADRs)

A running list of **Architecture Decision Records**. Each is a tiny, dated note: what we decided, why, and what we considered. New decisions append here as we build. Status: ‚úÖ Decided ¬∑ üü° Proposed (awaiting your confirm) ¬∑ ‚¨ú Open.

### ADR-001 ¬∑ ‚úÖ ¬∑ Monorepo with shared `contracts` + `core`
- **Context:** web, mobile, and API share domain shapes; they must not drift.
- **Decision:** one repo; `contracts` is the single source of truth for types; `core` holds framework-free domain logic; dependencies point inward.
- **Alternatives:** multi-repo + published packages (rejected: version-bump friction, drift).

### ADR-002 ¬∑ ‚úÖ ¬∑ Next.js (web) + NestJS (api)
- **Decision:** `apps/web` = Next.js App Router; `apps/api` = NestJS (HTTP + queue workers in one app).
- **Why:** you specified these; both are strong, TS-first, well-documented.

### ADR-003 ¬∑ ‚úÖ ¬∑ pnpm + Turborepo
- **Decision:** pnpm workspaces for linking, Turborepo for cached task running.
- **Alternatives:** npm/yarn workspaces (weaker); Nx (more powerful, heavier) ‚Äî revisit if we outgrow Turbo.

### ADR-004 ¬∑ ‚úÖ ¬∑ Zod-based contracts
- **Decision:** define shapes as Zod schemas, infer TS types, reuse the schema for runtime validation on both sides. (In *this* design-system repo the `contracts/` draft is import-free TS; Zod is added when lifted to `packages/contracts`.)
- **Alternatives:** plain TS interfaces (no runtime check); OpenAPI-first codegen (heavier).

### ADR-005 ¬∑ ‚úÖ ¬∑ Money = integer minor units, never floats
- **Decision:** store/compute money as integer cents + a currency code; format only at the UI edge.
- **Open sub-decision (ADR-006):** DB column type.

### ADR-006 ¬∑ ‚úÖ ¬∑ Money column = `bigint` integer cents
- **Decision:** store money as **integer minor units** in a `bigint` column + a currency code; all math in integer cents in `core`; format only at the UI edge. (Chose this over `numeric` + decimal lib for exact, simple integer math.)

### ADR-007 ¬∑ ‚úÖ ¬∑ PostgreSQL + Drizzle
- **Decision:** Postgres; Drizzle ORM (typed, SQL-close, great for learning).
- **Alternatives:** Prisma (more magic, codegen); raw SQL (more boilerplate).

### ADR-008 ¬∑ ‚úÖ ¬∑ BullMQ + Redis for the queue
- **Decision:** BullMQ on Redis for OCR, categorization, recurring, recap, notifications.
- **Alternatives:** `pg-boss` (Postgres-only, less infra) ‚Äî reconsider if we want to avoid Redis.

### ADR-009 ¬∑ ‚úÖ ¬∑ Auth.js + short-lived JWT to the API
- **Decision:** Auth.js (NextAuth) handles web sessions; the API trusts short-lived JWT access tokens (mobile uses the same token flow). We own the data and learn the mechanics.
- **Alternatives:** Clerk/Auth0 (hosted, faster, costs); custom Nest JWT (most work).

### ADR-010 ¬∑ ‚úÖ ¬∑ REST from NestJS (no tRPC for now)
- **Decision:** plain REST resources from Nest, serving both web and mobile. A typed client wrapper for web can come later if wanted; no tRPC for now.

### ADR-011 ¬∑ ‚úÖ ¬∑ AI/OCR providers & data posture
- **Decision:** all AI runs behind a provider interface in `@spendlio/ai` (see ADR-019). Default is a deterministic **offline engine** (rules-based categorization, mock OCR, grounded templated chat) ‚Äî zero external calls, fully testable with no key. The **live** adapter uses the **Vercel AI SDK** (`ai` + `@ai-sdk/anthropic`/`@ai-sdk/openai`), provider-agnostic: Anthropic Claude (`claude-opus-4-8`) when `ANTHROPIC_API_KEY` is set; OpenAI when `AI_PROVIDER=openai` + `OPENAI_API_KEY`. Receipt OCR = multimodal LLM via `generateObject`, validated against the contracts receipt schema. **Privacy posture:** sensitive financial data only enters prompts on the live path the user explicitly enables (a key is set); the offline path keeps everything local. Model output is always re-validated against contracts before use (the model is untrusted input).
- **Alternatives considered:** binding directly to a single vendor SDK (rejected ‚Äî no provider swap, weaker streaming/UI integration); dedicated OCR services (Textract / Document AI) ‚Äî revisit if LLM OCR accuracy proves insufficient.

### ADR-012 ¬∑ ‚úÖ ¬∑ Workers are a separate app (`apps/worker`); jobs split across packages
- **Decision:** workers run as their **own app** `apps/worker` (BullMQ consumers), importing `packages/{core,db,queue,storage,contracts}` and **nothing from `apps/api`**. Job name+payload schemas live in `packages/contracts`; the work in `packages/core`; queue wiring in `packages/queue`. Producers enqueue from `apps/api`.
- **Why:** the work lives in `core`, so the worker doesn't depend on the API ‚Äî a separate app deploys/scales/fails independently (spiky OCR/AI load). Revised from an earlier "run-mode of apps/api" after review. Full reasoning in `07-queues-jobs.md`.

### ADR-016 ¬∑ ‚úÖ (model) / ‚¨ú (rates provider) ¬∑ Multi-currency
- **Decision:** every amount stores its **original currency**; amounts are never raw-summed across currencies. **Two views:** native (per-account/-currency, exact, the "pesos/dollars tabs") and converted (the user's **base currency**, approximate, labeled "as of <date>"). Original amount+currency is the source of truth; transactions also store an **FX snapshot** to the base currency at entry time for stable historical totals. Minor-unit scale is **per-currency** (not always 2 decimals). Splits computed in the expense currency; balances tracked per currency.
- **Open:** rates provider (ECB/openexchangerates/‚Ä¶), refresh cadence, exact rounding rule.
- **Why / full model:** `12-currency-and-fx.md`.

### ADR-017 ¬∑ ‚úÖ (model) / üü° (tooling) ¬∑ Internationalization

UI strings come from ICU message catalogs in the apps (not `contracts`/`core`); store category **keys**, translate labels in the UI; format numbers/dates/money via `Intl`; use logical CSS for RTL. Proposed tooling: `next-intl` (web), `i18next` (mobile).

### ADR-018 ¬∑ ‚úÖ ¬∑ `@spendlio/contracts` entity shapes authored from the table list
- **Context:** step 03 told us to port the design-system project's `contracts/src/*` shapes, but **those source files were not bundled into this repo**. Only `money/enums/common/transaction/jobs` were given verbatim in `docs/build/03-contracts-package.md`.
- **Decision:** authored the remaining entities (`user`, `account`, `category`, `budget`+`BudgetStatus`, `receipt`+`ReceiptLineItem`, `split`: `Person`/`Group`/`SplitShare`/`Split`/`Settlement`/`Balance`, `recap`: `MonthlySummary`/`CategorySpend`) from the table list in `03-database.md` + the `transaction.ts` template ‚Äî minimal fields, consistent conventions (integer-cents money, `ownedEntity`/`baseEntity`, `Create*/Update*` DTOs). Specific choices: `user` carries `defaultCurrency` + separate `locale`/`timezone` (ADR-016/017); `category.userId` is nullable (null = built-in default); `Balance`/`BudgetStatus` are computed (not stored) and live in contracts; `CreateSplitInput` takes `participantIds`/`weights` and `core` computes the per-person `shares`; `receipt` stores an `imageKey` (storage key, not URL) + raw OCR payload. `recurring_rules` and `notifications` were intentionally deferred ‚Äî not in step 03's file list; they arrive with their features.
- **Alternatives:** block and ask for every field (rejected: unproductive; shapes are easy to extend and the DB step will reconcile). These shapes are provisional and may be refined when `core` (step 05) and the API (step 06) firm up the split/budget contracts.

### ADR-019 ¬∑ ‚úÖ ¬∑ AI work lives in `@spendlio/ai` (provider interface; offline default + AI-SDK live)
- **Context:** categorization, receipt OCR, and the assistant need an LLM SDK (Vercel AI SDK: `ai` + `@ai-sdk/anthropic`/`@ai-sdk/openai`). Golden Rule 1 forbids `core`/`contracts` importing any framework/SDK.
- **Decision:** all LLM-dependent work lives in a dedicated `@spendlio/ai` package behind an `LLMProvider` interface. The default is a deterministic **offline** engine (no key, no calls); the **live** adapter uses the AI SDK (Anthropic when `ANTHROPIC_API_KEY`; OpenAI when `AI_PROVIDER=openai` + `OPENAI_API_KEY`), model default `claude-opus-4-8`. The chat assistant is a **streaming request/response, not a queue job**: `streamChat()` ‚Üí `toUIMessageStreamResponse()` is served by `apps/api` and consumed by the web `useChat` hook. The assistant's tools return exact integer cents ‚Äî the model never does money math. Model output is re-validated against contracts before use. `@spendlio/ai` may depend on `core`/`db`; neither depends on `ai`. The worker imports `@spendlio/ai` for OCR/categorize/recap job work.
- **Alternatives:** binding directly to `@anthropic-ai/sdk` (rejected: vendor lock-in, no provider swap, weaker streaming/UI integration). Superseded the initial step-04 plan to use the official Anthropic SDK, at the user's request.

### ADR-020 ¬∑ ‚úÖ ¬∑ `fx_rates` table shape & money-row FK policy
- **Context:** multi-currency needs historical, auditable rates (`12-currency-and-fx.md`) and money rows must never silently disappear (Golden Rule 7).
- **Decision:** (a) `fx_rates(base, quote, date, rate)` is **global** (no `user_id`); `rate` is an exact decimal **string** (`varchar 32`, never float) with `UNIQUE(base,quote,date)`; FX math runs in integer minor units in `core` (round half-up to the target minor unit). (b) Transactions snapshot FX inline (`fx_base_currency` / `fx_base_amount` cents / `fx_rate` string / `fx_as_of` YYYY-MM-DD), null when original == base, so historical totals never drift. (c) FK `onDelete`: `user_id` cascades; money/cross-entity FKs (account, payer, person, group) **restrict**; pure-join rows cascade. (d) Cyclic refs among transactions/splits/receipts/recurring are plain-uuid soft-FKs to avoid circular DDL.
- **Consequence:** rates are reproducible and exact; deleting a user cleans up their data while a referenced account/person/group cannot be hard-deleted out from under money rows.

### ADR-021 ¬∑ ‚úÖ ¬∑ Split balances use the user-implicit viewpoint (model B)
- **Context:** `core.netForUser` needs a "me" id, but the schema has no `users`‚Üí`people` link / `is_me` flag, and the contracts `Balance` type treats the user as the implicit viewpoint with every `Person` a counterparty.
- **Decision:** `balancesSummary` scopes to splits owned by the user (`splits.user_id = userId`); the user is the implicit creditor; each `split_shares.personId` owes their amount; a **settled** settlement subtracts that amount from the person's debt; computed by direct per-person aggregation in exact integer cents. `splits.payerId` stays a required FK on insert but is unused by balance netting.
- **Consequence:** correct for the common "user paid" case (what the #10 proof + the live integration test exercise). Mixed-payer / "a friend paid me" splits (where the user owes) need a self-person row + bidirectional netting ‚Äî **deferred follow-up**.

### ADR-022 ¬∑ ‚úÖ ¬∑ AI live (AI-SDK) module is lazy-loaded, kept out of consumers' typecheck graph
- **Context:** `@spendlio/ai`'s barrel statically re-exported the live provider (`src/live`), which imports the Vercel AI SDK (`streamText`/`generateText`/`tool<‚Ä¶>` generics). Because consumers (`apps/api`, `apps/worker`) consume `@spendlio/ai` from TS source via tsconfig `paths`, importing the barrel dragged `src/live` into the consumer's tsc program ‚Äî and `apps/api` `tsc --noEmit` **OOM-crashed** (SIGABRT, ~303s, >6 GB) instantiating the AI SDK generic surface. Runtime was unaffected (`ts-node` transpileOnly).
- **Decision:** the barrel and `config.ts` must NOT statically reference `./live` or `@ai-sdk/*`. `getProvider()` stays sync; when a key is set it returns a `LazyLiveProvider` proxy that loads the real provider via a runtime `import()` with a **non-literal specifier** (`const LIVE_MODULE='./index'`), which tsc types as `any` and does not follow ‚Äî so consumers never instantiate the AI SDK types. Live-only env/model helpers live under `src/live/`. The package's own typecheck still checks `src/live` (kept type-safe); only consumers stop pulling it.
- **Consequence:** `apps/api typecheck` 303s+OOM ‚Üí **1.76s**; `apps/worker` 2.35s; `@spendlio/ai` self green (21 tests). Runtime identical (offline default; live on key; streaming preserved). Rule for future AI-SDK code: keep it under `src/live/` and out of the barrel.

### ADR-024 ¬∑ ‚úÖ ¬∑ Explicit self-person for model-B split balances (ADR-021 follow-up)
- **Context:** split balances treat the **payer** of a split as "you" and drop it from the result (`SplitsService.balances`). With no `users‚Üípeople` link, modeling "you paid, friends owe you" required either sacrificing a real friend as the payer (they vanish from balances) or fabricating one ‚Äî and the seed/Add-expense flow both hit this. `CreateExpenseAction` worked around it by passing `payerId = participantIds[0]`, which silently drops that first friend's debt (catastrophic for the common single-friend split).
- **Decision:** add an explicit **self-person** ‚Äî a `people` row with a new `is_self boolean` column (default `false`) representing the implicit "you". It is the payer/creditor viewpoint for model-B splits and is **excluded from `GET /people`** (`PeopleService.list` filters `is_self = false`) so it never pollutes people pickers, the People card, settle-up, or group membership. The seed creates one self-person per demo user and uses it as `payerId` (and a participant, so even-split remainder cents land correctly); friends then net out as debtors. Migration `0001` adds the column.
- **Scope / not done:** ~~`SplitsService.create` and `createExpenseAction` were **not** changed ‚Äî splits created through the Add-expense modal still pass `payerId = participantIds[0]` (pre-existing quirk). Fully routing user-created splits through the self-person (server-side payer override + `computeSplit` handling a payer outside `participantIds`) remains a follow-up.~~ ‚Äî ‚úÖ **resolved in ADR-028** (server-side self payer, participant-ownership IDOR check, core remainder fix, both balance readers made self-aware). This ADR resolved the balances/seed correctness and people-list pollution; ADR-028 completes the modal's split-create path.
- **Alternatives:** a `users.self_person_id` FK (heavier, same effect); an `is_me` flag on `splits` instead of `people` (doesn't solve the payer-FK requirement). Supersedes the "deferred" disposition in ADR-021 for everything except the modal split-create path.

### ADR-025 ¬∑ ‚úÖ ¬∑ Split Groups resource + synchronous per-person "Remind"
- **Context:** the canonical Split design (`docs/design-bundle/ui_kits/web/Split.jsx`) shows a **Groups** card (named groups of people with a net badge) and a per-person **Remind** action. The repo had `groups`/`group_members` tables and `Group`/`CreateGroupInput` contracts but no endpoint or reminder feature, so the Split page shipped People + balances + settle-up only.
- **Decision:** (a) a NestJS `groups` resource ‚Äî `GET /groups` returns each group with its `memberIds`; `POST /groups` (validated by `CreateGroupInput`) creates a group + `group_members`, rejecting any member that isn't one of the user's own people. No new contract or DB change (both already existed). **Group net is derived in the web layer** by summing the current balances of a group's members ‚Äî not stored, not a new endpoint (per-group attribution via `splits.groupId` is available but unused for now; documented in the page). (b) **Remind** is a synchronous `POST /people/:id/remind` that records an in-app `settle_reminder` **notification** for the user referencing the person ‚Äî reusing the existing `notifications` table (the same `settle_reminder` type the notify worker already knew). It is a direct user action, so it runs in-request rather than as a queue job (precedent: the streaming assistant, ADR-019). We have **no email/push transport** yet; actual delivery to the person lands when that infra exists.
- **Alternatives:** enqueue a `notify` job (rejected: its id-only `{userId,type}` payload can't reference a person and its deterministic jobId would dedupe all reminders into one); a no-op toast (rejected: leaves no record). Computing group net server-side in `core` (deferred: the web already holds balances; YAGNI).

### ADR-026 ¬∑ ‚úÖ ¬∑ Content-addressed receipt uploads (client SHA-256 + dedup + integrity)
- **Context:** receipt bytes upload **directly** to object storage via a presigned PUT ‚Äî the API never sees them (ADR-014). The flow used a random per-upload key with no dedup and no integrity check; re-uploading the same receipt created a new row and a fresh (paid) OCR call, and nothing verified the stored object matched what the client intended.
- **Decision:** the client computes a **SHA-256** of the file (Web Crypto, `crypto.subtle.digest`) before upload ‚Äî the only place that can hash, since bytes never reach our server. The hash flows through the existing 3-step flow without changing response shapes: (1) `presign(contentType, sha256)` derives a **content-addressed key** `receipts/<userId>/<sha256>.<ext>` (keys stay **per-user** so identical content across users never collides ‚Äî no cross-user dedup/privacy signal); (2) browser PUTs to it; (3) `register({ imageKey, sha256 })` **dedups** ‚Äî if the user already has a live receipt with that hash it returns the existing row and skips the OCR enqueue. The **worker re-hashes** the fetched bytes and refuses to OCR on mismatch (the client-asserted hash is untrusted until the one component that sees the bytes verifies it). `sha256` is stored on the `receipts` row (migration `0002`, nullable for legacy rows) with a non-unique `(userId, sha256)` lookup index. The `ocr` job payload stays id-only (`{ receiptId }`, Golden Rule 5).
- **Deliberately deferred:** a generic `files` table ‚Äî only receipts use storage today, so the `receipts` row *is* the file record (YAGNI). Revisit when a second uploader exists (avatars, statement imports, mobile). A **partial-unique** index `(userId, sha256) WHERE deleted_at IS NULL` for hard duplicate prevention under concurrent registers ‚Äî the service-level dedup SELECT covers the single-user case, and duplicate rows would be harmless anyway (same content-addressed key). 
- **Verified live:** content-addressed key, dedup (same hash ‚Üí same receipt id, no re-OCR), and integrity (a mismatched hash fails the job) all confirmed against the seeded stack.

### ADR-027 ¬∑ ‚úÖ ¬∑ Receipt review & approve ‚Üí expense (human-in-the-loop money)
- **Context:** OCR is best-effort and especially fragile across locales ‚Äî ARS receipts print `1.500,00` (dot=thousands, comma=decimals), so the model can emit the wrong scale ("wrong quantity of zeros"). Auto-committing OCR money is therefore unsafe, and there was no path from a receipt to a transaction anyway (the "create from receipt" action was omitted for lack of an endpoint).
- **Decision:** OCR output is a **suggestion the user verifies and corrects before it becomes money**. After upload the web routes to `/receipts/<id>`; once `parsed && !transactionId` the page shows an **editable review form** (merchant, date, total, currency, add/remove/edit line items with an "auto-sum to total" affordance, required category). **Approve** ‚Üí new **`POST /receipts/:id/confirm`** (validated by `ConfirmReceiptInput`) writes the corrected values back to the receipt **and** creates the linked expense (`amount = -total`, `source:'ocr'`, `receiptId`), setting `receipt.transactionId`. Re-confirming is rejected (400). **State model:** no enum change ‚Äî `transactionId != null` means "converted"; the page then shows a "Added to expenses ‚úì ‚Üí view in transactions" panel. **Money scaling:** the review form takes human MAJOR units and the server action converts to integer minor units via **per-currency `toMinorUnits(major, currency)`** (ARS=2, JPY/CLP=0, BHD=3) ‚Äî replacing the hardcoded `√ó100` for this path.
- **Deferred / noted:** ~~the pre-existing hardcoded `√ó100` in the other manual-entry actions~~ ‚Äî ‚úÖ fixed (all of `createTransactionAction`/`createExpenseAction`/`settleUpAction` now use per-currency `toMinorUnits`). AI-suggested category pre-fill (manual pick for v1). A per-transaction detail route (link to the `/transactions` list for now). The Add-expense modal is still USD-only (no currency picker), so its client-side `√ó100` is correct there.
- **Verified live:** parsed receipt ‚Üí confirm created `-150000` ARS (=$1,500.00) groceries expense linked to the receipt; re-confirm ‚Üí 400; `toMinorUnits` scales ARS/JPY/USD/BHD correctly. Spec: `docs/superpowers/specs/2026-06-15-receipt-review-approve-design.md`.

### ADR-028 ¬∑ ‚úÖ ¬∑ Route user-created splits through the self-person (resolves ADR-021/024 follow-up)
- **Context:** `SplitsService.balances()` treats a split's payer as "you" (skips its share, drops it from the net map). The Add-expense modal worked around the missing user‚Üíperson link by sending `payerId = participantIds[0]` ‚Äî a **real friend** ‚Äî so that friend's entire debt silently vanished (catastrophic for the common single-friend split). ADR-024 added the `is_self` person and fixed the seed/balances/people-list but explicitly deferred the modal's split-create path. A second balance reader also existed: the AI assistant's `netBalances` (`@spendlio/ai`) counted **every** share row (the opposite of `balances()`), so it already leaked a phantom "You" from the seed's self share.
- **Decision:** user-created splits route through the self-person **server-side** (model B). `CreateSplitInput` **drops the client `payerId`** (the payer is always you); `dto.total` is the **full expense**. `SplitsService.create` resolves the user's self-person (lazily creating one, made race-safe by a partial unique index), **validates participant ownership** (each must be one of the user's people and not the self-person ‚Äî the missing IDOR check), sets `payerId = self`, computes shares over `[self, ...participants]` with self absorbing the remainder, and **stores the self share** so rows reconcile to `total`. `core`'s `splitEven`/`splitPercent` no longer drop a cent when the payer isn't a participant, and assert `Œ£shares === total`. **Both** balance readers are now self-aware: `balances()` already skipped the payer; `netBalances` skips the self share and drops self from output (shipped together). `get()`/`list()` strip the self share (presentation). The web sends the full expense as `total` and no `payerId`.
- **Migration:** `0003` adds `people_one_self_per_user on people(user_id) where is_self`.
- **Supersedes:** the "modal split-create" deferral in ADR-024; resolves the ADR-021 follow-up in full.
- **Verification:** core split tests (payer-outside-participants, self remainder, every split sums exactly); `netBalances` self-skip unit + integration cases; assistant no longer reports a "You" balance. (DB/Redis-gated e2e to be run with the local stack up.)

### ADR-029 ¬∑ ‚úÖ ¬∑ Queue hardening + a real dead-letter queue
- **Context:** jobs were enqueued without payload validation (a bad payload poisoned the worker, then aged out of BullMQ's evictable fail set ‚Äî the de-facto "dead-letter view"). `notify`/`recurring` inserts were check-then-act with no DB constraint (retries/concurrent sweeps could double-insert money/notifications). `recap.month` was an unvalidated `z.string()` (a bad month produced NaN bounds and overwrote a good recap with zeros). The `ocr ‚Üí categorize` chain was dead for the real flow (categorize was enqueued off `receipt.transactionId`, null at OCR time; `confirm()`, which actually creates the transaction, never enqueued it).
- **Decision:** (a) `enqueue()` validates the payload against the contracts job schema **at the producer**. (b) DB-backed idempotency: a `notify` `dedupeKey` + partial unique index `(user_id, type, dedupe_key)` and a `recurring` partial unique index `(recurring_id, occurred_at)`, both with `onConflictDoNothing`. (c) `RecapJob.month` tightened to `/^\d{4}-\d{2}$/` + an invalid-range guard. (d) categorize is enqueued from `receipts.confirm()` after the transaction exists; OCR early-returns on an already-`parsed` receipt so a retry never re-calls the paid provider. (e) bounded provider timeouts (`AbortSignal.timeout`) + an OCR-worker `lockDuration` above that bound. (f) a **real dead-letter queue**: a durable `dead_letters` table the worker writes on a job's final failed attempt, routed to a single `alertFailure()` sink, with a `redriveDeadLetter()` re-enqueue helper. DLQ persistence lives in `apps/worker` (not `packages/queue`) so queue stays a BullMQ-only package per the layering rules.
- **Migration:** `0004` adds `notifications.dedupe_key` + its partial unique index, the `transactions` recurring partial unique index, and the `dead_letters` table.
- **Supersedes:** the "the failed set is the dead-letter view" framing in ADR-008 / `07-queues-jobs.md`.

### ADR-030 ¬∑ ‚úÖ ¬∑ Contracts own the presign shape; live receipt schema points at the contract
- **Context:** the presigned-upload response was a hand-copied Zod schema in `apps/web` mirroring `@spendlio/storage`'s `PresignedUpload` interface (drift waited to happen), and the live AI provider re-inlined the receipt OCR shape (`z.string().length(3)` for currency) instead of reusing the existing `ReceiptOcrResult`.
- **Decision:** `@spendlio/contracts` owns `PresignedUploadSchema` (+ `PresignedUpload` type); `@spendlio/storage` and the web edge both consume it (the web's mirror is deleted). The live provider's `generateObject` uses `ReceiptOcrResult` directly, removing the re-inline. Also: the pure recap/recurring math moved from `apps/worker/src/lib` into `@spendlio/core` (with unit tests), and `MeService`/`RecapsService` were extracted so those controllers stay thin.
- **Deferred:** moving the assistant tool-row types (`CategorySpend`/`BudgetLine`/`RecentTransaction`/`BalanceLine`) + `ReceiptOcrResult`/`CategorizeInput` wholesale into contracts ‚Äî they're the AI seam's own surface (ADR-022), and forcing the divergent `CategorySpend` (tool `amountCents` vs summary `amount`) into one shape couples two layers for little gain. Revisit if a non-AI consumer needs them.

### ADR-031 ¬∑ ‚úÖ ¬∑ Responsive layout via token-driven utility classes (not Tailwind / not CSS Modules)
- **Context:** the web app and `@spendlio/ui` style everything with **inline `style={{}}` objects**. Inline styles physically cannot express `@media`/`@container` queries or `:hover`/`:focus` pseudo-states, so the app shipped with **zero layout media queries** ‚Äî a fixed 248px sidebar that never collapses and 16 hardcoded multi-column grids with no mobile fallback. It is not responsive, and the same inability to express shared layout drove ~35‚Äì40% copy-pasted styling across pages. `styles.css` already documents the escape hatch (the "COMPONENT INTERACTIONS" block: *"the few things inline style can't express live here"*).
- **Decision:** keep the token-driven inline-style convention for per-instance look, and move **only layout + breakpoints** into a small, fixed vocabulary of utility classes in `@spendlio/ui/styles.css`, applied via the existing `cn()` helper. Per-instance values (a grid ratio, a gap) are passed inline as CSS custom properties (`style={{ '--spl-cols': '1fr 1.7fr' }}`) so the breakpoint logic stays in CSS while the value stays at the call site. The starter set: `.spl-stack`, `.spl-cluster`, `.spl-cards` (auto-fit, no breakpoint), `.spl-grid-asym`, `.spl-form-row`, `.spl-only-mobile`/`.spl-only-desktop`. Canonical breakpoint **768px** (token `--bp-md`; written literally in `@media` because conditions can't read CSS vars). Table‚Üîcard and similar swaps render **both** and let CSS pick (no JS `useMediaQuery`, no hydration mismatch); the only responsive JS is the shell's drawer toggle.
- **Alternatives:** **Tailwind** (rejected ‚Äî contradicts the token-driven posture, adds a monorepo build step, and would run two styling systems in parallel; every semantic token would need a theme mapping). **Per-component CSS Modules** (rejected ‚Äî a ~42-file rewrite away from the inline+token convention for marginal gain; the layout problems are concentrated in ~15 files, not 42).
- **Supersedes:** the implicit "desktop-only" assumption. Preserves the token vocabulary unchanged.

### ADR-032 ¬∑ ‚úÖ ¬∑ Feature-based front-end structure for `apps/web`
- **Context:** `apps/web` mixed routing, domain UI, shared composites, and server actions with no layering. Feature components were colocated under their route folder (`app/transactions/AddTransactionForm.tsx`), shared ones lived in a grab-bag `app/_components/`, and there was no mid-tier component layer between the `@spendlio/ui` primitives and the page files ‚Äî so forms, stat cards, category rows, and list rows were re-rolled per page.
- **Decision:** reorganize the web app into three explicit layers. **`packages/ui`** keeps app-agnostic visual primitives (reusable by the future mobile client). **`apps/web/components/`** holds shared cross-feature WEB composites (`layout/`, `form/`, `domain/`, `feedback/`). **`apps/web/features/<domain>/`** holds domain-specific UI + server actions + feature utils as `{components, hooks, lib}`. **`apps/web/app/`** becomes routing-only: thin `page.tsx` files that compose a feature view. `apps/web/lib/` (api/config/resources/safe) is unchanged.
- **Rule of thumb:** primitive (web+mobile-agnostic) ‚Üí `packages/ui`; web layout/composite ‚Üí `apps/web/components`; domain-specific ‚Üí `features/<domain>`.
- **Sequencing (done ‚Äî merged to `main`):** the move landed as a pure-`git mv` change (no behavior change); the genuinely-duplicated composites `FormField`/`FieldError` were extracted on top. The other proposed composites (`PageContainer`/`SectionCard`/`StatCard`/`CategoryBreakdownRow`/`ListRow`/`IconBadge`) were deliberately **not** built ‚Äî the category/stat/icon patterns proved too divergent across pages to share without an over-parameterised abstraction (simplicity-first). All 57 component files were then renamed PascalCase ‚Üí **kebab-case** (filenames kebab, exported symbols stay PascalCase). The `unicorn/filename-case` lint rule is **deferred** until a package actually runs ESLint ‚Äî none has a `lint` script today, so the rule would be inert (noted in `CLAUDE.md`).

### ADR-033 ¬∑ ‚úÖ ¬∑ Email OTP (Auth.js) + short-lived JWT to the API
- **Context:** ADR-009 decided "Auth.js + short-lived JWT to the API" but left the provider, the token mechanics, and first-sign-in provisioning open. Phase 5 needs to remove the dev `AuthGuard` (which trusts an `x-user-id` header and defaults to the seeded demo UUID) and the web's matching `DEMO_USER_ID` header injection (`apps/web/lib/api.ts` + the assistant proxy). Constraints: keep the **web ‚Üí API ‚Üí DB** layering (the web must not touch the DB), keep `contracts`/`core` framework-free, and keep every API route scoped by the *authenticated* `user_id` (golden rule #4, already honored in every service).
- **Decision:** sign-in is **email OTP** (6-digit code), not magic-link. **Auth.js (next-auth v5)** runs on the web with a **JWT session strategy and no DB adapter** (so no `accounts`/`sessions`/`verification_tokens` tables). The **API owns the OTP challenge**: public `POST /api/auth/otp/{request,verify}` endpoints (no guard), codes stored **HMAC-hashed in Redis** with a 10-min TTL, 5-attempt limit, single-use, 60s resend cooldown, and **no email enumeration** (`/request` always 200). First successful verify **provisions** by upserting `users` on the already-unique `email` ‚Äî **no migration**. Auth.js reaches the API via a **Credentials provider** (`authorize` ‚Üí verify endpoint); a second **dev-only Credentials provider** (`NODE_ENV !== 'production'`) logs in as the seeded demo user so local dev needs no email. Per server request the web **mints a short-lived HS256 JWT** (`jose`; `{ sub: user_id, iss: 'spendlio-web', aud: 'spendlio-api', exp: +5m }`, signed with a shared `API_JWT_SECRET`) and sends it as `Authorization: Bearer ‚Ä¶` from `'server-only'` code (it never reaches the browser); the **API guard verifies** it and sets `req.user = { id: sub }`. The `x-user-id` trust path is **removed entirely**. Email sends behind an **`EmailSender` interface** (golden rule #6): dev = console log; the concrete prod vendor is deferred to its own ADR (see open question #2). The live contract test swaps `x-user-id` for a Bearer JWT minted with the same secret.
- **Verification:** `pnpm typecheck` + `pnpm --filter @spendlio/ui test` + `pnpm --filter @spendlio/contracts test` + `pnpm --filter @spendlio/web build` stay green; manual: dev-login renders demo data, OTP sign-in provisions a fresh user scoped to their own data, and the API returns 401 to missing/invalid Bearer tokens.
- **Alternatives:** Auth.js native **email/magic-link provider** (rejected ‚Äî needs the DB adapter, putting the web on the DB and breaking the layering); **Auth.js Drizzle adapter** with DB sessions (rejected ‚Äî second DB writer + migration owner); **custom Nest credentials** with passwords (rejected ‚Äî hand-rolled hashing/reset, contradicts ADR-009); **OTP in a DB table** (rejected for now ‚Äî Redis TTL fits ephemeral codes with no migration); **OAuth (Google/GitHub)** (deferred ‚Äî purely additive later).
- **Deferred:** refresh tokens / mobile secure-storage; OAuth providers; MFA; Postgres RLS backstop; per-IP rate limiting on `/request`; OTP email as a queue job; RS256/JWKS instead of the shared HS256 secret; account self-management. Tracked in `08-auth-security.md`.
- **Design spec:** `docs/superpowers/specs/2026-06-16-phase-5-auth-otp-design.md`.

### ADR-034 ¬∑ ‚úÖ ¬∑ Retry a failed receipt scan + surface a friendly failure reason
- **Decision:** Denormalize a typed `failure_reason` code onto the `receipts` row (worker classifies the raw error via `core.classifyOcrFailure` on the final failed attempt; raw error stays in `dead_letters`) rather than joining `dead_letters` on read ‚Äî the receipt stays self-contained and cheap for list + detail, and `core` stays pure/testable. The web maps code ‚Üí friendly text; raw errors are never shown. Retry goes through a new `queue.requeue()` that **removes the existing deterministic job before re-enqueuing**, because BullMQ dedupes by `jobId` and `removeOnFail` keeps the failed job, so a plain re-`enqueue` is a no-op. Retry re-runs OCR on the same image; re-uploading a new photo is out of scope.

### ADR-035 ¬∑ ‚úÖ ¬∑ OCR suggests a spending category; reviewed category is authoritative
- **Decision:** The OCR vision model now infers a `CategoryKey` (`ReceiptOcrResult.category`, `.nullable()` so it stays OpenAI-strict-output-safe and the model can abstain) ‚Äî it has the richest context (merchant + line items), so this beats a second rules/LLM pass on merchant text. The category rides the existing `ocr` JSONB (no new column) and is surfaced via `ReceiptSchema.category` to pre-fill the review form. Because the user reviews/edits it before approving, `receipts.confirm()` no longer enqueues the `categorize` job ‚Äî the reviewed category wins and is not silently overwritten. The `categorize` job remains for non-receipt flows (manual/import/recurring). Constraining the suggestion to expense-only categories is deferred (the user catches the rare `income`/`transfer` misfire at review).

### ADR-036 ¬∑ ‚úÖ ¬∑ Real OTP email over SMTP (nodemailer) + Mailpit for local dev
- **Context:** ADR-033 shipped email OTP behind an `EmailSender` interface but deferred the concrete prod vendor; only `ConsoleEmailSender` (logs the code) existed, so a real user never received a code. "Next up ¬∑ A" in `PROGRESS.md`.
- **Decision:** Implement the deferred transport as a single **vendor-neutral `SmtpEmailSender`** (nodemailer), selected by `AuthModule` when `SMTP_HOST` is present ‚Äî else the dev `ConsoleEmailSender`. This mirrors `@spendlio/ai`'s offline-default / live-gated-on-env pattern, and SMTP (not a Resend/SES SDK) keeps local‚Üîprod a **config swap** (golden rule #6): local points at a **Mailpit** container added to `docker-compose.yml` (SMTP `:1025`, web inbox `:8025`, "$0 infra", so the SMTP path is exercised + verifiable locally); prod swaps `SMTP_*` creds to SES / Mailgun / Resend-SMTP with no code change. Auth is optional (omit when `SMTP_USER`/`SMTP_PASS` are unset ‚Äî Mailpit needs none); nodemailer timeouts are bounded to ~10s so a dead host can't hang `/request`. On send failure `OtpService.request()` **rolls back the Redis code + cooldown keys and returns 503** (not a bare 500) so the user can retry immediately; surfacing a transport error doesn't leak enumeration (it's unrelated to whether the account exists). The `/request` **`devCode` echo is gated on the transport's `exposesCode` capability** (true only for the console sender), not on `NODE_ENV` ‚Äî so a real SMTP send never leaks the code in the response, even in a non-prod/staging env that points `SMTP_HOST` at a real provider (an adversarial-review finding: the old `NODE_ENV`-only gate would have been an OTP bypass there). `secure` is derived from the port (465 ‚áí implicit TLS) when `SMTP_SECURE` is unset, so a 465 host can't silently connect in plaintext and stall.
- **Verification:** `pnpm --filter @spendlio/api typecheck` + `@spendlio/contracts` tests green; **live** (full stack + Mailpit): `/request` ‚Üí email lands in the Mailpit inbox with the code ‚Üí `/verify` provisions the user; SMTP transport **withholds** `devCode` from the response while the console transport still echoes it; `SMTP_HOST` unset ‚Üí console fallback (code logged); SMTP pointed at a dead port ‚Üí **503** + an immediate retry is accepted (cooldown was cleared). Reviewed by a 3-lens adversarial workflow (2 findings confirmed + fixed, 1 refuted).
- **Alternatives:** Resend / SES vendor SDK (rejected ‚Äî vendor lock at the impl layer + heavier setup; both expose SMTP anyway, so the interface stays portable); send OTP as a queue job (deferred ‚Äî OTP volume is trivial; synchronous send + user-retry suffices); React Email / templating (rejected ‚Äî inline strings, YAGNI).
- **Design spec:** `docs/superpowers/specs/2026-06-17-real-otp-email-smtp-design.md`.

### ADR-037 ¬∑ ‚úÖ ¬∑ Register-time receipt `imageKey` namespace guard (IDOR)
- **Context:** `POST /receipts` (register) trusted the client-asserted `imageKey` verbatim (`CreateReceiptInput.imageKey` was only `z.string().min(1)`), stored it, and `GET /receipts/:id/image-url` later presign-**downloads** `row.imageKey`. A caller could register a row whose `imageKey` points at **another user's** object (`receipts/<otherId>/<hash>.<ext>`) and read that blob back ‚Äî a cross-tenant IDOR. Latent in practice (keys embed an unguessable userId + sha256/uuid), but the trust gap was real: `presign` derives the key server-side, yet `register` re-asserts it with nothing rebinding the two. (Surfaced while reviewing the MinIO key/path logic; sibling to the ADR-028 IDOR pass, which hardened the presign *query* but not the register *key*.)
- **Decision:** Add a pure predicate `isOwnReceiptKey(key, userId, sha256?)` in `@spendlio/storage` ‚Äî co-located with `receiptKey`, since the module that defines the key format should own "is this key mine" ‚Äî and enforce it in `ReceiptsService.create()` before any DB write (throws **400**). Rule: the key must be a **single object directly under `receipts/<userId>/`** (no nested path / `.`/`..` traversal), and when a `sha256` is declared the basename must name that hash (so a stolen key can't be smuggled in under a different hash). Chose a unit-testable storage helper over an inline service check (the logic is pure + security-critical, and `@spendlio/storage` already runs Vitest) and over a Zod refine on the contract (the contract can't see the authenticated `userId`).
- **Verification:** 7 new `@spendlio/storage` unit tests (accept server-built + content-addressed keys; reject foreign prefix, traversal, nested, outside-root, sha256 mismatch, empty) ‚Äî 15/15 green; storage + api typecheck green; **live** (minted demo Bearer on a throwaway port): foreign-prefix / traversal / outside-root register ‚Üí 400 `imageKey is not in your namespace`, while a valid token still lists `200` ‚Äî and no rows are written (rejected pre-insert).
